✦ Quick Recap — How Streams Work

We've already covered how streams process data from various sources. Collections are the most common, but streams can also originate from arrays, files, generator functions, or even I/O channels. Regardless of the source, the processing model is always the same: data elements flow through a pipeline of intermediate operations and finish with a single terminal operation.

Source
Data origin

Collection, array, file, generator, I/O channel…

▸
Intermediate
Transform & filter

Reshape, select, reorder. Chainable — each returns a new Stream.

→ returns Stream<T>
▸
Terminal
Produce a result

Triggers execution of the entire pipeline. Does not return a Stream.

→ returns value / void

Intermediate Operations

An intermediate operation takes a stream in and hands a stream back out — that's what makes chaining possible. You can link as many as you need into a continuous pipeline. Here are the ones you've already seen:

map transform each element
filter keep elements matching a condition
sorted reorder elements
distinct remove duplicates
peek inspect without altering (debugging)
limit / skip truncate or skip elements

Terminal Operations

A terminal operation sits at the end of the pipeline and kicks off the actual processing. Nothing happens until a terminal operation is invoked — the entire pipeline is lazy. The result is never a stream; it's a value, a collection, or a side effect.

collect accumulate into a container
forEach perform an action on each element
reduce fold into a single value
count count elements
min / max find extremes
anyMatch / allMatch test a condition
findFirst / findAny retrieve a single element
toArray convert to an array
Remember: streams are lazy and single-use No computation runs until a terminal operation is called. And once a stream has been consumed by a terminal operation, it cannot be reused — you'll get an IllegalStateException if you try.

With these fundamentals in place, the natural next question is: what can we actually produce at the end of a pipeline? So far we've mostly used collect(Collectors.toList()) to gather results into a list — but that barely scratches the surface. The Collectors class offers dozens of powerful strategies for reshaping, aggregating, and transforming stream output. Let's take a look at the full toolkit before diving into examples.

1. Beyond toList() — The Collectors Toolkit

You already know .collect(Collectors.toList()). But Collectors is a factory with dozens of useful collectors. Today we'll focus on the ones you'll reach for most often.

1.1 — groupingBy

The workhorse of data reshaping. It transforms a flat list into a Map<K, List<V>> — think SQL's GROUP BY.

Collector<T, ?, Map<K, List<T>>> groupingBy(Function<T, K> classifier)
Example 1a
// Group students by major
Map<String, List<Student>> byMajor = students.stream()
    .collect(Collectors.groupingBy(Student::major));
{CS=[Alice, Bob, Frank], Math=[Charlie, Diana], Physics=[Eve, Grace]}
How groupingBy works
Key insight The classifier function extracts the map key from each element. Elements that produce the same key end up in the same list.

1.2 — Downstream Collectors

What if you don't want a List as the map value? The two-argument groupingBy accepts a downstream collector — a second collector that determines how to aggregate each group.

Example 1b — counting
// How many students in each major?
Map<String, Long> countByMajor = students.stream()
    .collect(Collectors.groupingBy(
        Student::major,
        Collectors.counting()
    ));
{CS=3, Math=2, Physics=2}
Example 1c — averagingDouble
// Average GPA per major
Map<String, Double> avgGpaByMajor = students.stream()
    .collect(Collectors.groupingBy(
        Student::major,
        Collectors.averagingDouble(Student::gpa)
    ));
{CS=3.3, Math=3.7, Physics=3.65}
Example 1d — mapping + joining
// Comma-separated names per major
Map<String, String> namesByMajor = students.stream()
    .collect(Collectors.groupingBy(
        Student::major,
        Collectors.mapping(
            Student::name,
            Collectors.joining(", ")
        )
    ));
{CS=Alice, Bob, Frank, Math=Charlie, Diana, Physics=Eve, Grace}
Pattern: composing downstream collectors Think of it as a pipeline inside the grouping: first mapping transforms each element, then joining aggregates the results. You can nest these arbitrarily deep.
Downstream collector composition — how nesting works

1.3 — partitioningBy

A special case of grouping that splits data into exactly two groups using a predicate — the map always has keys true and false.

Example 1e
// Split students into honors (GPA >= 3.5) and non-honors
Map<Boolean, List<Student>> honorsSplit = students.stream()
    .collect(Collectors.partitioningBy(s -> s.gpa() >= 3.5));

List<Student> honors    = honorsSplit.get(true);   // [Alice, Charlie, Diana, Eve, Grace]
List<Student> nonHonors = honorsSplit.get(false);  // [Bob, Frank]
When to use partitioningBy vs groupingBy Use partitioningBy when your classification is binary (pass/fail, above/below a threshold). It guarantees both keys exist in the map even if one group is empty. groupingBy only creates keys for groups that have members.

1.4 — toMap

Creates a Map from key/value extraction functions. Great for building lookup tables.

Example 1f
// Build a lookup: student name → GPA
Map<String, Double> gpaByName = students.stream()
    .collect(Collectors.toMap(
        Student::name,     // key extractor
        Student::gpa       // value extractor
    ));
{Alice=3.8, Bob=3.2, Charlie=3.9, Diana=3.5, Eve=3.7, Frank=2.9, Grace=3.6}
Duplicate key danger If two elements produce the same key, toMap throws IllegalStateException. Always provide a merge function when duplicates are possible: 
Example 1g — handling duplicate keys
// If two students share a major, keep the higher GPA
Map<String, Double> bestGpaByMajor = students.stream()
    .collect(Collectors.toMap(
        Student::major,
        Student::gpa,
        Double::max           // merge function for collisions
    ));
{CS=3.8, Math=3.9, Physics=3.7}

2. flatMap — Flattening Nested Data

Real-world data is rarely flat. Students have lists of courses, orders have lists of items, users have lists of roles. flatMap is how you "unwrap" one level of nesting.

2.1 — The Problem

What map() gives you (wrong shape)
// Get all enrolled course codes
List<List<String>> nested = students.stream()
    .map(Student::enrolledCourses)
    .collect(Collectors.toList());
// Result: [[CS101, CS201, MATH301], [CS101, CS301], ...]
// We wanted a flat list, not a list of lists!
map() vs flatMap() — visual comparison

2.2 — The Solution

Stream<R> flatMap(Function<T, Stream<R>> mapper)

flatMap maps each element to a stream, then concatenates all those streams into one.

Example 2a
// All unique course codes across all students
List<String> allCourses = students.stream()
    .flatMap(s -> s.enrolledCourses().stream())
    .distinct()
    .sorted()
    .collect(Collectors.toList());
[CS101, CS201, CS301, MATH201, MATH301, PHYS101, PHYS201]
Mental model map transforms 1 element → 1 element (shape stays the same). flatMap transforms 1 element → 0..N elements (flattens one level).

2.3 — flatMap for Data Enrichment

A common pattern: create a pair for every combination of student and course, essentially "exploding" each student into multiple rows.

Example 2b
// Produce a record for every (student, course) pair
record Enrollment(String studentName, String courseCode) {}

List<Enrollment> enrollments = students.stream()
    .flatMap(s -> s.enrolledCourses().stream()
        .map(c -> new Enrollment(s.name(), c))
    )
    .collect(Collectors.toList());
[(Alice, CS101), (Alice, CS201), (Alice, MATH301), (Bob, CS101), (Bob, CS301), ...]

2.4 — Combining flatMap with Collectors

Now the two sections come together. What if we want to know which students are in each course?

Example 2c — flatMap + groupingBy
// Build a reverse index: course code → list of student names
Map<String, List<String>> courseRoster = students.stream()
    .flatMap(s -> s.enrolledCourses().stream()
        .map(c -> Map.entry(c, s.name()))
    )
    .collect(Collectors.groupingBy(
        Map.Entry::getKey,
        Collectors.mapping(Map.Entry::getValue, Collectors.toList())
    ));
{CS101=[Alice, Bob, Charlie, Frank], CS201=[Alice, Eve], CS301=[Bob], MATH201=[Charlie, Diana, Grace], MATH301=[Alice, Charlie, Eve], PHYS101=[Diana, Eve, Grace], PHYS201=[Grace]}
The Map.entry trick When you need to carry two pieces of data through a flatMap, wrapping them in a Map.Entry (or a record) is a clean, allocation-light approach. This avoids defining throwaway classes.
