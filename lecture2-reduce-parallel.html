1. reduce — The Universal Aggregator

Every terminal operation that produces a single result (count, sum, min, max) can be expressed as a reduce. Understanding reduce means understanding what's really happening under the hood.

1.1 — The Two-Argument reduce

Optional<T> reduce(BinaryOperator<T> accumulator)

Takes pairs of elements and combines them down to one. Returns Optional because the stream might be empty.

Example 1a — sum of all GPAs
Optional<Double> totalGpa = students.stream()
    .map(Student::gpa)
    .reduce(Double::sum);
// totalGpa = Optional[24.6]

Example 1b — longest name
Optional<String> longest = students.stream()
    .map(Student::name)
    .reduce((a, b) -> a.length() >= b.length() ? a : b);
// longest = Optional[Charlie]

1.2 — The Three-Argument reduce (with identity)

T reduce(T identity, BinaryOperator<T> accumulator)

Provides a starting value. Now the result is a plain T, not an Optional, because if the stream is empty we simply return the identity.

Example 1c
// Total number of course enrollments across all students
int totalEnrollments = students.stream()
    .map(s -> s.enrolledCourses().size())
    .reduce(0, Integer::sum);
// totalEnrollments = 17
The identity contract The identity value must be the "zero" of the operation: accumulator(identity, x) == x for all x. For addition → 0. For multiplication → 1. For string concatenation → "". For max → Integer.MIN_VALUE.

1.3 — Reduce to a Different Type (Three-Arg with Combiner)

What if the result type is different from the element type? The three-argument form handles this:

<U> U reduce(U identity, BiFunction<U, T, U> accumulator, BinaryOperator<U> combiner)
Example 1d — build a comma-separated string from names
String allNames = students.stream()
    .reduce(
        "",                                           // identity
        (str, student) ->                              // accumulator: String × Student → String
            str.isEmpty() ? student.name()
                         : str + ", " + student.name(),
        (s1, s2) -> s1.isEmpty() ? s2                  // combiner: String × String → String
                   : s2.isEmpty() ? s1
                   : s1 + ", " + s2
    );
Alice, Bob, Charlie, Diana, Eve, Frank, Grace
When do you need the combiner? The combiner is only used in parallel streams to merge partial results from different threads. But you must provide it even for sequential streams when using this overload. Make sure it's consistent with the accumulator — we'll see why in Section 2.

1.4 — Practical reduce: Building a Summary Object

Example 1e — accumulate into a custom summary
record MajorStats(int count, double totalGpa) {
    MajorStats add(Student s) {
        return new MajorStats(count + 1, totalGpa + s.gpa());
    }
    MajorStats merge(MajorStats other) {
        return new MajorStats(count + other.count, totalGpa + other.totalGpa);
    }
    double averageGpa() { return count == 0 ? 0 : totalGpa / count; }
}

MajorStats csStats = students.stream()
    .filter(s -> s.major().equals("CS"))
    .reduce(
        new MajorStats(0, 0.0),  // identity
        MajorStats::add,              // accumulator
        MajorStats::merge             // combiner
    );

System.out.println(csStats);           // MajorStats[count=3, totalGpa=9.9]
System.out.println(csStats.averageGpa()); // 3.3
reduce vs collect reduce produces immutable intermediate values — each step creates a new object. collect (via Collector) uses a mutable container. For building up collections, strings, or large objects, collect is more efficient. Use reduce when combining values (sums, products, statistics) or when immutability is important.

2. Parallel Streams

Java streams can execute on multiple threads with a single method call. But "easy to write" doesn't mean "always a good idea."

2.1 — Making a Stream Parallel

Example 2a — two ways to go parallel
// Option 1: from the source
students.parallelStream()
    .filter(s -> s.gpa() > 3.5)
    .map(Student::name)
    .collect(Collectors.toList());

// Option 2: convert mid-pipeline
students.stream()
    .parallel()
    .filter(s -> s.gpa() > 3.5)
    .map(Student::name)
    .collect(Collectors.toList());
Under the hood Parallel streams use the ForkJoinPool.commonPool(). The source is split (via Spliterator), each chunk is processed on a separate thread, and results are combined. The combining step is why reduce's combiner and collector's combiner exist.

2.2 — The Rules for Correct Parallelism

For a parallel stream to produce the same result as a sequential one, your operations must satisfy three properties:

Property Meaning Example violation
Stateless Lambdas must not read/write shared mutable state Incrementing an external counter inside map
Non-interfering Don't modify the source during execution Calling list.remove() inside forEach
Associative (a ⊕ b) ⊕ c == a ⊕ (b ⊕ c) Subtraction: (10-5)-2 ≠ 10-(5-2)

2.3 — Classic Bug: Shared Mutable State

Example 2b — WRONG: race condition
// DON'T DO THIS — data race with parallel stream
List<String> results = new ArrayList<>();

students.parallelStream()
    .filter(s -> s.gpa() > 3.0)
    .forEach(s -> results.add(s.name()));  // ⚠️ ArrayList is NOT thread-safe
// Possible: missing elements, ArrayIndexOutOfBoundsException, corrupted list

Example 2c — CORRECT: let the collector handle it
List<String> results = students.parallelStream()
    .filter(s -> s.gpa() > 3.0)
    .map(Student::name)
    .collect(Collectors.toList());  // ✅ Thread-safe: each thread has its own list
The golden rule Never accumulate into a shared mutable container from a stream — use collect. This applies to sequential streams too (for clarity), but with parallel streams it causes actual concurrency bugs.

2.4 — Ordering Guarantees

Example 2d — forEachOrdered preserves encounter order
// forEach with parallel: order is non-deterministic
students.parallelStream().map(Student::name)
    .forEach(System.out::println);
// Could print: Eve, Alice, Frank, Diana, Bob, Grace, Charlie

// forEachOrdered: preserves original list order (but loses some parallelism)
students.parallelStream().map(Student::name)
    .forEachOrdered(System.out::println);
// Always: Alice, Bob, Charlie, Diana, Eve, Frank, Grace
collect preserves order Even in parallel streams, .collect(Collectors.toList()) preserves encounter order. The framework handles the merge correctly. You only lose ordering with forEach and unordered collectors.

3. When Does Parallel Actually Help?

Parallel streams have overhead: splitting, thread scheduling, combining. That overhead is only worth it when the per-element work or element count is large enough.

3.1 — Rules of Thumb

Factor Favors Parallel Favors Sequential
Data size 10,000+ elements < 1,000 elements
Per-element cost Expensive (I/O, computation) Trivial (field access, comparison)
Source splittability ArrayList, arrays, IntRange LinkedList, Stream.iterate, I/O streams
Terminal operation reduce, collect, toArray forEachOrdered, findFirst (order-dependent)

3.2 — Benchmarking Parallel vs Sequential

Example 3a — demonstrating with a CPU-bound task
// Simulate expensive computation per student
private static double expensiveGpaCalculation(Student s) {
    // Simulate work with a spin-wait (DON'T do this in production)
    long start = System.nanoTime();
    while (System.nanoTime() - start < 1_000_000) {} // ~1ms per student
    return s.gpa() * 4.0 / 4.0;
}

// With a large list (e.g., 10,000 students), parallel shows real gains:
long start = System.nanoTime();

double avg = bigStudentList.parallelStream()      // swap with .stream() to compare
    .mapToDouble(s -> expensiveGpaCalculation(s))
    .average()
    .orElse(0);

long elapsed = System.nanoTime() - start;
System.out.printf("Avg: %.2f in %d ms%n", avg, elapsed / 1_000_000);
Never micro-benchmark with System.nanoTime alone For real performance testing, use JMH (Java Microbenchmark Harness). The example above is for classroom demonstration only. JVM warmup, JIT compilation, and garbage collection all affect results.

3.3 — The reduce Combiner in Action

Now we can see why reduce has a combiner. In parallel, each thread reduces its chunk independently, then the combiner merges the partial results:

Example 3b — parallel reduce with combiner
// Thread 1 reduces: [Alice, Bob, Charlie]   → MajorStats(3, 10.9)
// Thread 2 reduces: [Diana, Eve, Frank, Grace] → MajorStats(4, 13.7)
// Combiner merges:  MajorStats(3, 10.9).merge(MajorStats(4, 13.7))
//                 = MajorStats(7, 24.6)

MajorStats allStats = students.parallelStream()
    .reduce(
        new MajorStats(0, 0.0),
        MajorStats::add,     // accumulator: each thread builds its own
        MajorStats::merge    // combiner: merges results from different threads
    );
Combiner consistency rule The combiner must produce the same result as sequential accumulation. Formally: combiner(u, accumulator(identity, t)) == accumulator(u, t). If this doesn't hold, sequential and parallel will give different results.

3.4 — Quick Decision Flowchart

Ask yourself these questions before calling .parallel():

Decision Process
1. Is the data source large (>10K elements)?
   └─ No  → stay sequential
   └─ Yes → continue

2. Is the per-element operation expensive?
   └─ No  → probably stay sequential
   └─ Yes → continue

3. Is the source easily splittable? (ArrayList, array, range)
   └─ No  → stay sequential
   └─ Yes → continue

4. Are all lambdas stateless and non-interfering?
   └─ No  → FIX THIS FIRST, then reconsider
   └─ Yes → go parallel, then benchmark to confirm
